Release Engineering

Why Implement a Managed Release Cycle (MRC)

  The purpose of implementing a regular release cycle is to increase the stability of the production environment and provide better visibility into exactly what is going to change and when. An effective release management process allows IT to improve the speed and quality of changes to the production environment and to better organize the allocation of dedicated resources committed to oversee the integration of changes to our environment.

 As part of this process change all deployments that are not urgent should be shifted toward the Managed Release Cycle. Project work that requires an exception to the process for some reason (e.g. vendor schedules, weekend only upgrades, etc.) should be brought to the attention of the Release Management Team ( ReleaseManagement@Acadian-asset.com ) for scheduling.   

 Included with initiative to move to a managed deployment cycle and the recent organizational changes to IT development teams, we have realigned the managers who can Manager Approve business side FootPrint tickets. Going forward the approvals will be done by Jim Soper, Sarath Akkineni, and John King as they are th;e managers closest to the business projects and development processes.

Exceptions to the MRC process:

  • The following Footprints will NOT Be subject ot the MRC process:
    • Any fix for an Urgent issue that is affecting our ability to do business
    • Any fix to data quality that is likely to cause inaccuracies or errors with our processes
    • New User creation, additional user permissions, or application access
  • The Data - Data Modification/Updates Footprint category should be used only for Footprints that are correcting a data quality issue . Missing, incorrect, or duplicate data that needs to be corrected ASAP due to potential business impact and can be corrected with a SQL script. Footprints that are changing application behavior but require nothing but a SQL script should not be set to Data - Data Modification/Updates.

Key Points:

  • Any Footprint which causes an adverse impact to the production environment will be rolled back and set to "Assigned" status. These tickets will need to go through the normal release cycle process again to become candidates for the next scheduled MRC deployment.
  • Some Footprints approved for MRC deployment may be deployed after the standard MRC release day if there is a business constraint.
  • Whether an exception to MRC or not, all Footprints should receive some measure of QA review. This can be accomplished through formal QA by that team or as a peer review but a second set of eyes should verify that everything is in order before the Footprint is Manager Approved

What is the Release Schedule?

  • Monday Afternoon - Deployment Candidate pipeline Meeting
    • Overview of the tickets in the pipeline
    • Establish Level of Effort (LOE) for deployment
    • LOE Status
      • Large – Dev Complete by EOD Tuesday
      • Medium – Dev Complete by EOD Wednesday
      • Small – Dev Complete by EOD Thursday
    • Footprints that are not set to Dev Complete status by the assigned cutoff will need to wait until the next MRC
  • Friday - 5PM
    • All tickets must be QA Approved for release on Tuesday
  • Friday - 6PM
    • An automated report will be sent to all of IT with the list of all tickets that qualify for the following Tues MRC deployment
  • Monday
    • PSG and DBOps will ensure that all tickets are ready for deployment on Monday.
    • Once verified a spreadsheet named "Release Candidates ddMMMyy.xlsx" will be created and posted to this site
  • Tuesday
    • PSG, DBOps, and SrvOps will meet and discuss the deployment schedule for the day
    • An email will be sent from Release Management when deployment starts
    • A follow-up email will be sent when deployments have been completed
    • An updated spreadsheet, "released tickets ddMMMyy.xlxs", will be posted to the SharePoint site.

What if there is an office closure due to Holiday?

  • In the event of a Monday office closure all key dates remain unchanged except the release date will be on Weds
  • In the event of a Friday office closure all key dates remain the same.
  • All other scheduled office closures will be handled ad-hoc and communicated by Release Management to all involved two weeks prior to the event.

What is the Release Process?

 We already have a Footprints process that we are accustomed to using and as we move to a Managed Release Cycle we are leveraging that existing procedures with some minor changes. With these changes the new workflow is as follows:

  • Users, BSA/ PMs and developers are encouraged to enter the Target Date field.
    • In the case of an urgent hotfix or data update script that must be run ASAP this date will be the same day.
    • If the Footprint is not an urgent data update or urgent fix then the Target Date should be set to the appropriate Tuesday.
  • If there are any steps for DBOps to execute then DB Review Required has to be selected; no exceptions.
  • DBOps will notify PSG if there are any steps for them and wait for their "OK" before DBA Approving. (This is required due to a lack of a "PSG Approval" step in the Footprints workflow)
  • QA will wait until after DBA Approval before setting QA Approved.
  • QA Approval must be complete by 5:00 pm on Fridays for any ticket to be deployed on the following Tuesday.
  • At 6:00 pm on Fridays, a report of the eligible tickets will be generated, e-mailed to all of IT. Once reviewed and annotated, it will be uploaded to the SharePoint site here. **Only the PSG and DBOps Managers will edit the release management spreadsheet/report.
  • Everyone with tickets that are expected to be released should review this list to ensure a ticket is not missed.
  • Managers will Manager Approve as their schedule permits. When they review tickets, where necessary, they will add or update the Target Date field before clicking the approval button.
  • Footprints are released by the Release Management team members according to their Target Dates and relative priorities.

What to do if a ticket is an emergency?

 In the event that there is an immediate need to correct an issue in the production environment the Footprint should be marked with priority of Urgent. The existing release process will take effect and Release Management staff will be paged to make them aware of the issue. Urgent Footprints are treated as a "drop what you're doing" situation so please keep this in mind when using this priority setting.

What to do if deploying a ticket will prevent an emergency?

  There will be times when a condition exists that we know is going to become an emergency; e.g. bad/missing data, incorrect results appearing on a report, etc. These hotfix situations should have the business reason the ticket needs to go out of cycle in the description.
